## Features Implemented

### 1. Admin User Management Panel

**Backend — `backend/app/routers/admin.py` (new file)**

A new FastAPI router mounted at `/api/admin` and protected by `get_current_superuser` (HTTP 403 for non-admins). Endpoints:

| Method | Path | Description |
|---|---|---|
| `GET` | `/api/admin/users` | List all users, newest first |
| `PATCH` | `/api/admin/users/{id}` | Update role, active status, password, or name |
| `DELETE` | `/api/admin/users/{id}` | Permanently delete a user account |

Safety rules enforced server-side:
- An admin cannot demote themselves (`is_superuser = false` on own account is rejected).
- An admin cannot delete their own account.
- Email uniqueness is checked on update.
- New passwords must be at least 8 characters.

**Frontend — `frontend/src/pages/AdminUsers.tsx` (new file)**

A full admin panel page with:
- **Stats row**: Total users / Active users / Admin count.
- **Searchable user table**: Filter by username, email, or full name.
- **Inline action buttons** per row: toggle active/inactive, grant/revoke admin role, reset password (modal), delete (confirm dialog).
- Self-action protection: toggle-admin and delete buttons are disabled for the currently logged-in user.

**Sidebar — `frontend/src/components/Sidebar.tsx`**

An "Administration" nav group with a "User Management" link is conditionally rendered only when `user.is_superuser === true`. Non-admin users see no trace of the admin section.

**Routing — `frontend/src/App.tsx`**

Route `/admin/users` registered and wrapped in `ProtectedRoute`.

### 2. PDF Report Export

**Backend — `backend/app/routers/admin.py`**

Two endpoints (GET for simple cases, POST to avoid URL length limits for long analysis text):

| Method | Path | Description |
|---|---|---|
| `GET` | `/api/admin/flight-tests/{id}/report.pdf` | Generate PDF (analysis in query param) |
| `POST` | `/api/admin/flight-tests/{id}/report.pdf` | Generate PDF (analysis in request body) |

The PDF is built with **reportlab** and streamed as a download. Structure:

1. **Header**: FTIAS logo text + "Flight Test Interactive Analysis Suite" subtitle.
2. **Metadata block**: Flight Test ID, Test Name, Aircraft Type, Test Date, Generated By (user's full name or username), Generated At (UTC timestamp).
3. **Description** (if present on the flight test record).
4. **Section 1 — AI Analysis**: The full analysis text, markdown-stripped for clean PDF rendering.
5. **Annex A — Parameter Statistics**: A formatted table with Parameter, Unit, Min, Max, Mean, Std Dev, and Sample Count for all parameters in the flight test.
6. **Footer note**: Auto-generation disclaimer.

`reportlab>=4.0` was added to `backend/requirements.txt`.

**Frontend — `frontend/src/pages/FlightTestDetail.tsx`**

Two new buttons appear in the AI Analysis panel **after** an analysis has been generated, and **only for superusers**:

- **Download PDF**: Calls the POST endpoint, receives a Blob, triggers a browser download with a descriptive filename (`FTIAS_Report_{TestName}_{Date}.pdf`).
- **Print**: Calls the same endpoint, injects the PDF into a hidden `<iframe>`, and triggers `window.print()` on the iframe for browser print/save.

The `AIAnalysisPanel` sub-component was refactored to accept a `toast` prop from the parent `FlightTestDetail` component so PDF export feedback (success/error toasts) renders in the correct `ToastContainer`.

## Deployment Note

`reportlab` was installed in the running container for immediate use:

```powershell
docker exec ftias-backend pip install "reportlab>=4.0"
```

It is now also in `requirements.txt`, so the next `docker compose up -d --build backend` will bake it into the image permanently.
